team-harness

An agent harness for Claude Code. Turns the chat into a Spec-Driven Development pipeline with mandatory human gates and full state captured as files so any session can resume from where the last one left off.

{{VERSION}} MIT

Install

Step 1 — Add the marketplace
/plugin marketplace add valianx/team-harness
Step 2 — Install the plugin
/plugin install th
Step 3 — Configure and verify
/th:setup

Run these three commands inside Claude Code. /th:setup configures the Memory MCP server, context7 API key, and logs mode, then writes everything into ~/.claude/. Restart Claude Code after install. Full reference in docs/install.md.

What you get

After install, 17 specialized agents and 32 slash commands are registered in your ~/.claude/. The system runs a full Spec-Driven Development pipeline: intake, architecture design, implementation, tester/QA verification, security review, and delivery — all coordinated through workspaces files that persist across sessions.

The front door is the th:orchestrator agent. Open Claude Code and type:

@th:orchestrator give me the work plan for this task: <description>

The th:orchestrator opens a pipeline, stops for mandatory human approvals at each stage gate, and picks up exactly where it left off after a restart via @th:orchestrator recover <feature>.

Pipeline at a glance

plan it
Stage 1 — Analysis

Architect designs the solution, QA defines acceptance criteria, plan-reviewer audits shape. A STAGE-GATE stops the pipeline for human review before any code is written.

implement it
Stage 2 — Implementation

Implementer writes production code. Tester, QA, and security agents verify in parallel. An adversarial reviewer runs a pre-delivery audit on security-sensitive changes. Iteration loops run automatically until all gates pass.

ship it
Stage 3 — Delivery

Delivery agent writes the changelog, bumps the version, commits, and opens the PR. A final STAGE-GATE asks for push approval. State is captured in the Knowledge Graph for future tasks.

Agents

th:orchestrator hub

Central hub for all development workflows. Routes tasks through the full pipeline (architect → implementer → verify → delivery) with parallel test+validate and iteration loops. Also handles direct modes from standalone skills. Manages workspaces as the shared board between agents.

architect planning

Designs, evolves, and reviews software architecture for any project type (backend, frontend, or fullstack). Focuses on maintainability, security, performance, and accessibility. Produces architecture proposals, risk assessments, migration strategies, and technology research reports — never code.

implementer implementation

Implements features by writing production code based on architecture proposals and acceptance criteria from workspaces. Follows project conventions, writes clean code, and reports what was built. Does not design architecture, write tests, or create documentation.

tester validation

Designs and implements test suites for any project type (backend, frontend, or fullstack). Adapts to the project's test framework, ensures proper isolation, mocks external dependencies, and validates business logic, user interactions, and accessibility.

qa validation

Validates implementations against acceptance criteria and defines AC for features when invoked standalone. Produces validation reports — never code.

security validation

Performs comprehensive security audits on backend and frontend projects. Evaluates against OWASP Top 10, CWE Top 25, ASVS, and SANS Top 25. Detects vulnerabilities, hardcoded secrets, insecure configurations, auth flaws, and injection risks. Does not implement fixes or modify source code.

plan-reviewer validation

Read-only auditor of Stage 1 analysis artifacts. Enforces plan-shape rules: audits the Delivery Grouping (default all-tasks-one-PR, or temporal-prod-reasoned groups), per-task acceptance criteria in Given/When/Then format, consolidated documents, cross-references, and service-identity coherence. Emits pass/concerns/fail verdict before STAGE-GATE-1.

delivery delivery

Documents a completed feature, updates CHANGELOG and OpenAPI (if applicable), bumps the project version, creates a feature branch, commits, and pushes. Updates CLAUDE.md memory, project docs/ knowledge base, and README.md.

reviewer delivery

Reviews pull requests on GitHub. Analyzes code quality, security, performance, and best practices. Leaves detailed review comments and approves or requests changes.

agent-builder utilities

Designs and creates new Claude Code agents and slash commands. Applies best practices for system prompts, context management, memory, tool scoping, model selection, and output protocols. Always runs /lint after writing files.

init utilities

Bootstraps Claude Code in any repository (backend, frontend, or fullstack). Discovers the tech stack, generates a CLAUDE.md with golden commands and subagent orchestration, and creates a CHANGELOG.md if missing.

translator utilities

Discovers user-facing strings in any frontend codebase, builds a product-aware glossary, sets up i18n infrastructure, extracts strings into locale files, translates to target language, and replaces hardcoded strings with i18n keys. Works with any frontend framework.

gcp-cost-analyzer utilities

Analyzes GCP infrastructure costs and resource utilization across all accessible projects. Inventories resources via Cloud Asset Inventory and gcloud CLI, pulls billing data, fetches Recommender API suggestions, and produces a structured cost report with prioritized optimization recommendations. Does not modify or delete any GCP resources.

diagrammer diagrams

Generates Excalidraw diagrams from architect analysis. Reads the analysis, follows the excalidraw-diagram skill methodology, generates the .excalidraw JSON section-by-section, runs a render-validate loop until the diagram passes quality checks. Does not analyze codebases, write code, tests, or documentation.

d2-diagrammer diagrams

Generates D2 diagrams from architect analysis. Reads the analysis, follows the d2-diagram skill methodology, generates the .d2 source incrementally, validates with d2 fmt + compile, and reports back. Does not analyze codebases, write code, tests, or documentation.

likec4-diagrammer diagrams

Generates LikeC4 architecture diagrams from architect analysis. Reads the analysis, follows the likec4-diagram skill methodology, generates the .c4 DSL incrementally, validates with CLI, and reports back. Does not analyze codebases, write code, tests, or documentation.

Requirements

  • gh CLI (recommended) — GitHub CLI for full integration with /issue, /review-pr, and /deliver. When absent, skills fall back to curl or operator-paste paths. Install at cli.github.com.
  • context7 API key — for library docs retrieval. The installer prompts for it or reads CONTEXT7_API_KEY from the environment. Get one at context7.com.
  • Memory MCP URL — public URL of a running MCP-compatible server (e.g., context-harness-mcp deployed to Railway, Render, Fly, or Docker). The installer prompts for it or reads MEMORY_MCP_URL from the environment. No default — empty input is rejected. See docs/install.md for details.
