### Added

- Branch-aware `git push` gating in the dev-guard hook: a push to a single, non-default branch on the `origin` remote — with no force flag, no `+refspec`/`--mirror` prefix, no tag ref, and no delete form — now resolves to `permissionDecision: "allow"` without an interactive prompt. Every other push shape (default-branch destination, multi-refspec, `--mirror`/`--all`, non-`origin` remote, tag push, force by flag or by `+refspec`, branch delete) still resolves to `ask`, fail-closed.
- `gh pr create` opt-in autogate: a new `autogate.pr_create` key (default off) in `~/.claude/.team-harness.json`, read by the dev-guard, allows PR creation without a prompt once explicitly enabled by the operator; the tests-before-PR floor in prepublish-guard is a separate hook and is not bypassed by this opt-in (`deny` still wins over `allow`).
- Distributed read-only permission allowlist: `/th:setup` §3a and the orchestrator's Phase 0a Step 1g now also offer a scoped set of inert Bash commands (`git status/diff/log/show/rev-parse/branch --list/worktree list`, `ls`, `cat`, `rg`, `grep`), prefix-safe read-only `gh` verbs (`gh pr view/list`, `gh issue view/list`), `gh auth switch`, `mcp__memory__*` allow-rules, and `additionalDirectories` for the workspace/vault path — all under the existing gated Y/n confirmation with blast-radius disclosure.
- Batched, aliased GraphQL review-thread disposition: `agents/_shared/gh-fallback.md` gains a "Tier B — batched review disposition (aliased mutation)" section that composes all pending reply/resolve mutations into a single aliased `gh api graphql` request (reply aliases ordered before resolve aliases), with mandatory full-payload preview before the gated call and per-alias partial-failure reporting. The existing per-thread single-op sections remain the documented fallback.
- Single-PR release path: a `version.d/.release-cut` marker (or a `release-cut: vX.Y.Z` commit trailer), recognized by prepublish-guard on a feature branch when all three version sites are bumped and match, routes to the existing release-path logic instead of denying — folding the version bump, `changelog.d/` fragment assembly, and marker into the feature PR ahead of STAGE-GATE-3. The three-site version invariant is preserved, not relaxed: a missing or mismatched site still denies, and a marker whose content fails strict SEMVER parsing always denies.

### Changed

- The dev-guard's contract statement is reconciled across all nine documented sites (`docs/dev-mode.md`, `output-styles/developer-mode.md`, `docs/how-it-works.md`, `skills/setup/SKILL.md`, `skills/setup/managed-blocks/orchestrator-dispatch-rule.md`, `CLAUDE.md` §5, `docs/knowledge.md`, both `hooks/adapters/dev-guard.*.yaml`) from "every covered action asks unconditionally" to "fires unconditionally and gates by destination" — the hook still evaluates every covered action every time; only the decision for the non-default-branch push subset changes to `allow`.
- `/th:release`'s Step 3 (manual tag push) is degraded to verify-only; `tag-sync.yml` remains the sole deterministic tag authority (fires on `push:main` touching `.claude-plugin/plugin.json`), unchanged in logic.
- `.github/workflows/test.yml` adopts a conditional-noop gate-job pattern for version/changelog-only release diffs, avoiding the required-check-pending trap that a `paths-ignore` filter would introduce.

### Security

- Closed the positive-grammar push recognizer against the #18312 allow-rule-overrides-hook precedence: the allowlisted push shape is a single, explicitly-enumerated safe form (single refspec, destination extracted from the right side of the last colon, resolved on the payload cwd, remote matched to `origin` by name) — every unrecognized shape (multi-refspec, `--mirror`/`--all`, non-`origin` remote or URL, tag ref, delete form) fails closed to `ask`, never `allow`.
- Force-push detection is hardened on two independent axes: by flag (`--force`, `--force-with-lease`, `-f` — also independently denied by `policy-block`) and by refspec shape (`+refspec` prefix, `--mirror`) which `policy-block`'s flag-only detection does not cover; the dev-guard self-covers the second axis.
- The distributed read-only allowlist (WI-2/WI-3) is verified disjoint from the dev-guard's own outward-action catalogue by a new catalogue-driven test (`tests/test_permission_disjointness.py`) that derives the outward-action patterns directly from `hooks/ts/bodies/dev-guard.ts` and asserts no allow-prefix is a prefix of any outward-action sample, with a canary asserting the test does not pass vacuously. No form of `gh api` (including `gh api graphql`) is ever included in the allowlist, since no `gh api` prefix is disjoint from its own mutating forms.
- Accepted residual (NB-8): the dev-guard's default-branch resolution trusts `origin/HEAD` for repos with a non-standard default branch; the operator's Option-A decision accepts this residual, with `git remote set-head origin -a` documented as the recovery path when `origin/HEAD` drifts from the actual default.
