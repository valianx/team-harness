# Testing Reference: a11y tests

> Loaded on demand by the `tester` agent via the Reference Router.
> Stack-agnostic principles first, then per-stack sections keyed `## <stack>`.

## Principles (stack-agnostic)

- Accessibility tests verify that **the application is usable by people with disabilities** — auditing against WCAG criteria via automated tools (axe) and supplemented by manual keyboard / screen-reader checks.
- Automated axe scans catch the tractable class of violations (missing ARIA labels, low color contrast, invalid ARIA roles, unlabeled form fields). They do NOT replace manual verification for complex interactions (carousel, modal focus trap, drag-and-drop).
- **axe-clean is an acceptance criterion, not optional.** A component or page with axe violations is considered incomplete — it blocks merge, not just a warning.
- Two layers of axe coverage complement each other:
  - **Per-component** (via Storybook a11y addon): runs axe in a real browser per story. Catches component-level violations early, during development.
  - **Per-page** (via `@axe-core/playwright` in E2E): catches violations that only emerge when the full page is assembled — focus order, landmark structure, reading-order relationships.
- Anti-patterns: running axe only in jsdom (contrast and layout-dependent rules produce false results); treating axe violations as "nice to have"; skipping keyboard-navigation testing for interactive components.

## react-nextjs

Stack: Next.js App Router + TypeScript + Storybook 10.x a11y addon + `@axe-core/playwright`

### Layer 1: per-story axe via Storybook a11y addon

Configuration in `.storybook/vitest.setup.ts` and `preview.tsx` (see ui-component.md for the full setup). Key lines:

```typescript
// .storybook/vitest.setup.ts
import * as a11yAddonAnnotations from '@storybook/addon-a11y/preview'
const annotations = setProjectAnnotations([previewAnnotations, a11yAddonAnnotations])
beforeAll(annotations.beforeAll)
```

```tsx
// .storybook/preview.tsx — make axe violations fail the test, not just warn
parameters: { a11y: { test: 'error' } }
```

With this configuration, every story runs axe automatically under addon-vitest. No per-component addition needed. The check runs in a real browser (Chromium via Playwright), so contrast and layout-dependent rules produce correct results.

To exclude a known false-positive for a specific story:

```tsx
export const KnownFalsePositive: Story = {
  parameters: {
    a11y: {
      config: {
        rules: [{ id: 'color-contrast', enabled: false }],  // document the reason in a comment
      },
    },
  },
}
```

Only disable a rule for a story when the violation is a confirmed false-positive (documented). Do NOT disable rules to pass CI.

**No-Storybook path (Vitest Browser Mode without Storybook):** if the repo uses `vitest-browser-react` but has no Storybook, run axe-core directly against the rendered container inside the browser-mode test rather than falling back to jsdom axe (which this file already warns produces false results for contrast and layout rules):

```bash
npm install -D axe-core   # required for the No-Storybook path below
```

```typescript
// src/components/Modal/Modal.browser.test.tsx
import { it, expect, afterEach } from 'vitest'
import { render, cleanup } from 'vitest-browser-react'
import axe from 'axe-core'
import { Modal } from './Modal'

afterEach(() => cleanup())

it('is axe-clean when open', async () => {
  const { container } = await render(<Modal open title="Confirm action">Content</Modal>)
  const results = await axe.run(container)
  expect(results.violations).toEqual([])
})
```

This runs in a real browser context (Chromium via `@vitest/browser-playwright`), so contrast and layout-dependent axe rules produce correct results — the same fidelity as the Storybook a11y addon path. See `browser-mode.md` for setup.

### Layer 2: per-page axe via `@axe-core/playwright`

```typescript
// tests/e2e/a11y.spec.ts
import { test, expect } from '@playwright/test'
import { AxeBuilder } from '@axe-core/playwright'

test('home page is axe-clean', async ({ page }) => {
  await page.goto('/')
  const results = await new AxeBuilder({ page }).analyze()
  expect(results.violations).toEqual([])
})

test('checkout page is axe-clean', async ({ page }) => {
  await page.goto('/cart')
  const results = await new AxeBuilder({ page }).analyze()
  expect(results.violations).toEqual([])
})
```

Include page-level axe as part of any flow E2E test, not only in standalone a11y tests:

```typescript
test('checkout completes and page is axe-clean', async ({ page }) => {
  await page.goto('/cart')
  // functional assertions ...
  const results = await new AxeBuilder({ page }).analyze()
  expect(results.violations).toEqual([])
})
```

### Scope of automated coverage

| What axe catches automatically | What requires manual verification |
|-------------------------------|----------------------------------|
| Missing `alt` text | Reading order and visual flow |
| Missing form labels | Complex keyboard patterns (drag-and-drop, date pickers) |
| Insufficient color contrast | Screen-reader announcement timing |
| Invalid ARIA roles / attributes | Focus trap correctness in modals |
| Missing landmark structure | Cognitive load and plain-language compliance |
| Unlabeled buttons / links | — |

Report gaps discovered in manual review in `03-testing.md` alongside the automated results.

### CI wiring

Storybook a11y tests run as part of `vitest run` (the `storybook` project). Playwright a11y tests run as part of `playwright test`. Both block merge when `a11y.test: 'error'` is set and violations are found.

```yaml
# CI sketch — unit + stories (incl. axe per story)
- run: npm run test            # vitest: unit + storybook (incl. axe)
# E2E (incl. page-level axe)
- run: npm run test:e2e        # playwright: flows + page axe
```
