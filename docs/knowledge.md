# Knowledge Base
<!-- Project knowledge that agents read before working -->

- [pattern] Self-describing task list: `02-task-list.md` carries a `Status:` field per PR (`pending | in-progress | verified | merged | blocked`) and AC checkboxes flipped to `- [x]` on PASS — readers don't have to cross-reference `00-state.md` or `04-validation.md` to know progress.
- [pattern] Bidirectional cross-references for split PRs: base PR carries `Cleanup PR:` with rationale (what stays alive, condition for cleanup merge, risk if cleaned together); secondary PR carries `Base PR:` back-reference. The closed-list label `Split reason: coexistence window` alone is not enough.
- [pattern] Phase 1.6 plan-review is inviolable: `01-plan-review.md` must exist with a `## Verdict` line before STAGE-GATE-1 is emitted to the user. When the harness refuses a nested `Task(subagent_type='plan-reviewer')`, the orchestrator runs the 5-rule audit inline using `agents/plan-reviewer.md` as the procedure spec — never escalates to the user without the audit. Subagent and inline executions share the same max-3 iteration budget.
- [constraint] Post-STAGE-GATE-1 writes to `02-task-list.md` are limited to: the `Status:` field on a PR header (orchestrator transitions to in-progress/verified/blocked; delivery transitions to merged) and AC checkbox flips `- [ ]` → `- [x]` by qa on PASS. Files, AC text, dependencies, Cleanup PR / Base PR, Title, Branch, Notes are frozen.
- [constraint] Agent frontmatter (`agents/*.md`) MUST be valid YAML. An unquoted `": "` (colon-space) inside a `description:` value silently breaks parsing and Claude Code drops the agent from the registered `subagent_type` list with no error surfaced. `tests/test_agent_frontmatter.py` enforces this — runs via `uv run --with PyYAML python` and parses every agent file.
- [stack] As of feat/go-installer, the installer binary is Go (stdlib-only, CGO_ENABLED=0, cross-compiled). `bin/install.py` is deprecated (one-release fallback). Bootstrap wrappers (`bin/install.sh`, `bin/install.ps1`) download the correct binary from the latest GH Release — they no longer require `uv` or Python on the host.
- [pattern] KG passive capture on delivery: the `delivery` agent (Step 11.5 of `agents/delivery.md`) calls Memory MCP `create_nodes` once per completed task with `nodeType: "process-insight"`, synthesising one node from the workspaces + CHANGELOG entry. The call is best-effort — if the Memory MCP server is unreachable or no reusable learning surfaced, the step logs the skip and continues. This builds the team's institutional knowledge automatically without operator curation. v0.5.0+.
- [decision] Hook notifications fire **only when the user must take an action** (default preset `ultra-quiet` in `hooks/config.json`): `Notification` matcher `idle_prompt` + `PreToolUse` policy gate, nothing else. The previous `permission_prompt` matcher and blanket `PostToolUseFailure *` were removed because they fire while work continues (especially under `skipDangerousModePermissionPrompt: true` and during MCP flutter) and accumulate as look-alike toasts in the Windows Action Center / system tray without action required. `hooks/README.md` exposes louder presets (`default-quiet`, `noisy`) for users who want them — but the default is silence-by-default. Drives the "notifications must be actionable" contract for the whole `hooks/` distribution.
- [decision] Curl one-liner install via `go:embed` + GitHub Pages: installer binary is self-contained (`//go:embed agents skills hooks` at repo root level — embed paths must be in/below the source file's directory); bootstrap scripts (`bin/install.{sh,ps1,cmd}`) use the deterministic `releases/latest/download/` URL without calling the GitHub API (removes rate-limit exposure + brittle JSON parse); scripts are published to `https://valianx.github.io/team-harness/install.{sh,ps1,cmd}` via `.github/workflows/pages.yml` triggered on `release: published` only. Contributors testing local edits use `go run ./cmd/install` — `//go:embed` snapshots working-tree bytes at compile time.
- [decision] orchestrator stage-end toasts: the orchestrator fires 4 native OS notifications per pipeline at stage boundaries (Phase 1.6 close → Stage 1, Phase 2 last-PR close → Stage 2, Phase 3.6 last-PR close → Stage 3, Phase 4.5 close → Stage 4). Always fires regardless of autonomy mode. Goes through wrapper `hooks/notify-stage.sh` that detects OS via `$OSTYPE`/`uname` and routes to the existing `notify-{os}.sh`. Idempotent via `stage.notify` event lookup in `00-execution-events.jsonl` (local mode) or `00-execution-events.md` (obsidian mode). Deliberately bypasses the ultra-quiet hook preset (PR #26) — the preset governs Claude Code hook events; the orchestrator calls the wrapper directly. Remove `~/.claude/hooks/notify-stage.sh` post-install to silence.
- [pattern] gh-fallback graceful degradation: `agents/_shared/gh-fallback.md` is the single source of truth for degradation when `gh` CLI is absent. Detection probe → Tier A (curl reads) → Tier B (curl writes or operator paste with `blocked-manual-push`) → Tier D (project board skip). `//go:embed all:agents` includes the `_shared/` directory in the installer binary. v2.10.0+.
- [pattern] GitHub Actions re-review reminder via `assets/scaffolds/team-harness-rereview.yml`: triggers on `pull_request.synchronize`, compares `review.commit_id` with `pull_request.head.sha` to detect stale reviews, posts a comment nudging the operator to re-run `/review-pr`. Stateless — all state in GitHub. Scaffolded by `/init --scaffold-rereview-workflow`. v2.10.0+.
- [decision] Dual-mode workspaces: `logs-mode` in `.team-harness.json` controls whether pipeline output goes to local `./workspaces/` or an Obsidian vault. orchestrator resolves path once, passes to agents. Agents are mode-unaware.
- [decision] Workspace folders include date prefix: `{YYYY-MM-DD}_{feature-name}/` in both modes. Prevents collisions and enables chronological sorting.
- [decision] Human-first document format: all workspaces use `## Review Summary` + `## Technical Detail` layout. Review Summary uses Obsidian callouts (`> [!decision]`, `> [!risk]`, `> [!change]`).
- [pattern] Cross-platform skill specs use prose branching, not shell conditionals: skill `.md` files are agent instructions, not executable scripts — platform-conditional behavior is expressed as "if Windows, use X; if macOS/Linux, use Y" prose so the agent interprets at runtime. Example: `skills/update/SKILL.md`.
- [pattern] orchestrator state template includes `docs_root` and `logs_mode` for recovery: both fields are written to `00-state.md` at boot (Step 1c) so any agent resuming after context compaction can resolve all file paths without re-interrogating the manifest. Example: `agents/orchestrator.md` lines 371-372.
- [constraint] orchestrator Phase Checklist is a guardrail, not a suggestion: each phase must be marked `[x]` on completion or `[~skipped: reason]` for authorized skips before the next phase begins. An unmarked gap is a contract violation. Documented at `agents/orchestrator.md` lines 374-423.
- [decision] Execution events dual-format: in obsidian mode the orchestrator writes `00-execution-events.md` (YAML frontmatter + `# Execution Events` heading + ` ```jsonl ` code fence wrapping each JSON line); in local mode it writes `00-execution-events.jsonl` (unchanged). Skills `/trace` and `/status` detect both formats automatically (check `.md` first, fall back to `.jsonl`). The `.md` variant strips frontmatter and fence via `sed` before piping to `jq`. Pipeline crash leaves the fence unclosed; Obsidian renders it correctly. No migration of existing `.jsonl` traces.
- [decision] Merge/push guard: the orchestrator refuses to merge a PR or push to remote until Phase 3 (Verify) is `[x]` for that PR AND STAGE-GATE-3 is `[x]`. Operator instructions like "merge them" do not override — requires explicit "skip verification".
- [decision] orchestrator runs as subagent, not inline: inline execution (top-level Claude reads the contract from a file) weakened gate enforcement — stages were skipped because "fulfill user request" outweighed "follow contract read from file." Subagent mode loads the contract as system prompt, making it a stronger constraint. The boot probe's false "Task unavailable" message was cosmetic; Task always worked.
- [decision] Research→implement reclassification contract: when operator chooses "implement" after a research pipeline, the orchestrator reclassifies to `refactor`/`feature`, emits a `pipeline.reclassify` event, and re-enters the full pipeline at Phase 0b with all gates mandatory. Closes the gap where research pipelines could transition to implementation without verification.
- [decision] Operator language detection and propagation: the orchestrator detects `operator_language` (ISO 639-1) from the first operator message at Phase 0a intake. The code is stored in `00-state.md` and propagated to every agent dispatch prompt as `Operator language: {code}. Write workspaces prose in this language; structural elements stay English.` Fallback is `en` for ambiguous or short messages. Recoverable from `00-state.md` on resume; legacy pipelines lacking the field default to `en`.
- [decision] Plugin distribution via Claude Code marketplace: plugin name `th`, all skills namespaced as `/th:skill-name`. Custom marketplace at `valianx/team-harness`. Go installer maintained as alternative for offline/CI/low-cost mode. Migration guide at `docs/plugin-migration.md`.
- [decision] Skills restructured to directory format: all skills use `skills/<name>/SKILL.md` (compatible with both plugin auto-discovery and Go installer). Flat `.md` files deprecated. The installer copies skill directories recursively; the plugin auto-discovers all `SKILL.md` files under `skills/`.
