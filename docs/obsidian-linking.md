# Obsidian Work-Log Interlinking — Normative Contract

> This document is the authoritative specification for the three-tier MOC/wikilink scaffold
> generated by `agents/delivery.md` § Step 11.6. All implementation details in the delivery
> agent follow this contract. When in doubt, this document is the source of truth.

---

## Three-Tier Topology

Each obsidian-mode pipeline run produces (or regenerates) three deterministically-named files:

```
{worklogs_root}/_MOC-work-logs.md                              top MOC    → links each repo MOC
{worklogs_root}/{repo}/_MOC-{repo}.md                          repo MOC   → links each feature index + stray repo-root docs
{worklogs_root}/{repo}/{feature_dir}/{feature_dir}.md          feature index → links that run's durable stage docs
```

Where:
- `{worklogs_root}` = `{logs-path}/{logs-subfolder}` (e.g. `/vault/work-logs`)
- `{repo}` = the repository name (e.g. `team-harness`)
- `{feature_dir}` = the date-prefixed feature folder name (e.g. `2026-06-06_obsidian-worklog-interlinking`)

**The feature hub is a dedicated index note named after its folder** — NOT `00-state.md`. Rationale: Obsidian labels graph nodes by filename. Naming every index `00-state` or `_index` would produce an unreadable wall of identical graph dots. The folder-named index (`feature_dir/feature_dir.md`) gives each hub node a unique, meaningful label.

## Link Form

All cross-note links use **vault-relative path wikilinks** with forward slashes and a display alias. The `.md` extension is omitted for `.md` targets; non-`.md` targets keep their extension.

```
[[{logs_subfolder}/{repo}/{feature_dir}/{basename_no_ext}|{alias}]]
```

Example:
```
[[work-logs/team-harness/2026-06-06_obsidian-worklog-interlinking/01-plan|Plan: obsidian-worklog-interlinking]]
```

**Never use bare basename wikilinks** (`[[01-plan]]`). Bare basenames collide across feature folders; path-qualified links are globally unique because the date-prefixed `feature_dir` segment is unique per run.

**Forward slashes only.** Obsidian's wikilink convention uses forward slashes on every OS. Backslashes (Windows FS) must never appear inside a wikilink.

## H1-Label Fallback Chain

For each file being linked, derive its display alias using this ordered fallback:

1. **H1 heading:** read the first ~60 lines of the file; if a line matches `^# (.+)$` (a single `#` not `##`), use the captured text.
2. **Frontmatter `title:`:** if a frontmatter block (`---`…`---`) is present and contains a `title:` key, use its value.
3. **Humanized filename:** strip the extension, replace every run of `[-_]` with a single space, collapse whitespace, trim.

After deriving the raw label, apply `escape_alias(s)`:
- Remove `[` and `]`
- Replace `|` with `/`
- Replace any CR or LF with a space
- Collapse repeated spaces; trim
- Truncate to 120 characters

**Never use the raw filename as the alias when an H1 exists.** Labels like `Pr 08 Ux` are useless; the H1 `UX Validation — PR-08 (...)` conveys the topic.

## Knowledge-Only Allowlist

When scanning a feature folder for docs to link, the scan recurses one level into the `research/` subfolder (mirroring the `sketches/` precedent) and **includes only** files whose basename matches the knowledge allowlist:

| Included (knowledge-tier) | Reason |
|---|---|
| `research/00-research.md` (and `research/00-research*.md`) | Research and spike knowledge-tier doc — the pre-design study that informs the plan |
| `01-plan.md` (and `01-plan*.md`) | The consolidated design and decision record — the primary knowledge artifact |
| `01-root-cause.md` (and `01-root-cause*.md`) | Bug-fix flow knowledge-tier doc — the research-equivalent artifact for fix pipelines (fix flows have no `00-research`) |

**Everything else is excluded:**
- Process/verification docs under `reviews/`: `reviews/04-validation.md`, `reviews/04-security.md`, `reviews/01-ux-review.md`, `reviews/04-ux-validation.md`, `reviews/04-adversary.md`, `reviews/04-review.md`, `reviews/04-internal-review.md`
- Process/verification docs at the feature root: `02-implementation.md`, `02-documentation.md`, `03-testing.md`, `03-regression-tests.md`, `05-diagram.*`, `00-acceptance-criteria.md`
- Plumbing: `00-state.md`, `00-execution-events.md`, `00-execution-events.jsonl`, `session.json`
- The feature-index note itself (`{feature_dir}.md`) — excluded to avoid self-reference

Rationale: `01-plan.md` is the consolidated knowledge record (it embeds Stage-1 ratification, plan-review, risks, and security design); research + plan + root-cause carry all durable knowledge. Process and verification docs are useful detail during a pipeline run but clutter the knowledge graph hub with ephemeral content.

## Stray Repo-Root Docs

Some repositories carry `.md` files directly under `{worklogs_root}/{repo}/` outside any feature folder (cross-repo reports, CONFIG/VISION notes, etc.). The repo MOC links these under a `## Repo Docs` section so nothing is orphaned. The `## Repo Docs` section is omitted entirely when there are no stray docs.

---

## Idempotent Whole-Tree Regeneration

Each of the three index/MOC files is **fully rewritten** (whole-content overwrite) from a deterministic scan on every delivery run. No append-with-markers, no entry-marker bookkeeping.

Consequence: re-running delivery (or `/th:recover`) converges to identical file content — no duplicate entries, no stale links. Operator hand-edits inside a generated index/MOC will be overwritten on the next delivery. The auto-managed banner (`> Auto-generated … do not hand-edit.`) in every generated file makes this behaviour visible.

---

## Forward-Only Reconciliation

The feature-index step (Step 11.6.1) generates the feature index for **the current run only**. The repo MOC step (Step 11.6.2) and top MOC step (Step 11.6.3) scan for index notes and repo MOCs that **already exist on disk** — they never create index notes for historical feature folders that lack one.

This means:
- Historical folders the operator indexed manually are discovered and kept (their `<d>/<d>.md` exists).
- Historical folders with no index note are left untouched — they are not auto-created.
- Forward runs accumulate: each new delivery adds its feature index, then the MOCs are regenerated to include it.

## Local-Mode and Tier-0 No-Ops

| Condition | Behavior |
|---|---|
| `logs_mode == "local"` | Step 11.6 is a no-op. Log `obsidian_interlink: skipped: local-mode`. |
| `docs_root` does not exist (Tier-0 fix / no-workspace run) | Step 11.6 is a no-op. Log `obsidian_interlink: skipped: no-workspace`. |

The index/MOC files are written to the Obsidian vault (`{logs-path}/{logs-subfolder}/...`), never to the repo working tree. They are never staged or committed.

## Sanitization and Alias Escaping

### Path component sanitization

Before composing any filesystem path or wikilink, validate `repo` and `feature_dir` against `[A-Za-z0-9._-]+`. Reject any component that:
- Contains `..`, `/`, or `\`
- Contains any character outside `[A-Za-z0-9._-]`

Additionally, if `feature_dir` contains a 10-character date prefix, validate it matches `^\d{4}-\d{2}-\d{2}`.

On any sanitization failure: log `operation.failed` (`detail: "obsidian-interlink-sanitize"`) and skip the entire step. Never write a partial file.

### Alias escaping

Aliases derived from H1 text may contain characters that break wikilink syntax. Apply `escape_alias(s)` as specified in the H1-Label Fallback Chain section above. This prevents markdown/wikilink injection via crafted H1 text (`]]`, `\|`, `[[`, newlines).

## Known Obsidian Limitation — Graph Dot Labels

**Obsidian's floating graph always labels dots by filename, never by the link alias.** Even with well-crafted H1-derived aliases, the graph dot text shows the filename.

**Accepted trade-off:** the meaningful navigation is in the rendered link lists inside each index/MOC note (which DO show aliases), and the hub/MOC filenames (`_MOC-{repo}`, `{feature_dir}`) are themselves unique and human-legible navigation entry points. Renaming timestamped docs to "fix" the dot label is invasive and rejected.

**Mitigation:** naming convention ensures the filenames are already meaningful:
- `_MOC-team-harness` → clearly identifies the repo hub
- `2026-06-06_obsidian-worklog-interlinking` → date-prefixed, unique per run

---

## Cross-Platform Note

All wikilink composition uses forward slashes regardless of the OS. The delivery agent uses the Read/Write/Glob tools (not shell commands) for index/MOC file operations, so there is no shell-portability concern.
